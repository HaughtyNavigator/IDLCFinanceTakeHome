## 2026-07-22
- Requirements audit against the company brief: all functional/technical items covered, but found `docker compose up` hard-failed on a fresh clone — `env_file: .env` is mandatory in Compose and `.env` is (correctly) gitignored, so an evaluator cloning the repo hit `env file ... not found`. Verified by running compose in an empty directory.
- Fix: `env_file` now `required: false` plus an `environment:` passthrough, so the app runs from `.env` OR from `GEMINI_API_KEY=... docker compose up` with no file at all. Verified all three cases (fresh clone parses, shell var wins, existing `.env` still resolves and is not clobbered by the empty default).
- Security review of that change: no new exposure — key stays gitignored, stays out of the image, still runtime-injected; `docker inspect` reveals env identically under either mechanism. Real delta is fail-closed becoming fail-open, addressed below.
- Fix: missing key previously surfaced as 502 "AI service temporarily unavailable" — misleading, reads as an upstream outage rather than an operator error. Added `GeminiConfigurationError(GeminiServiceError)` and a dedicated handler returning 503 naming the variable, plus a critical log line at startup via a lifespan hook.
- Decision: startup check logs and continues rather than exiting. Compose sets `restart: unless-stopped`, so exiting on boot would crash-loop — worse for an evaluator than the failure it replaces.
- Cleanup: untracked `this is not an image` (stray corrupt-upload fixture) and `.claude/settings.local.json` (machine-local paths); added `.claude/` to `.gitignore`.
- Tests now 17 (added missing-key 503 case); black/isort clean. `PROJECT.md` and `CLAUDE (2).md` intentionally kept in the repo at the user's request as evidence of process.
- User report: hazy images produce hallucinated field values — the model reconstructs plausible text instead of leaving fields null. Existing prompt line ("return null if not confident") proved insufficient on its own.
- Considered three options: (1) model-reported per-field confidence, (2) token logprobs (`response_logprobs` exists in the SDK — real uncertainty, but hard to map to fields under structured output), (3) self-consistency across N samples (strongest signal, since hallucinations vary between runs while true readings are stable, but N× cost). Chose (1) for cost and explainability; (3) remains the upgrade path.
- Implemented per-field confidence: `FieldConfidence` enum + `FieldConfidences` model on the internal Gemini schema; `to_public()` nulls any field rated below `MIN_FIELD_CONFIDENCE` (default `medium`). Public 10-field contract unchanged — confidence is internal and stripped, same pattern as `is_nid`/`readability_issue`.
- Prompt rewritten with explicit transcribe-don't-reconstruct rules and per-field rating criteria, including tie-break guidance (unsure between two levels -> pick the lower).
- Design decision: `FieldConfidences` defaults to HIGH so a response omitting the object falls back to prior behavior rather than nulling all ten fields; unrecognized `MIN_FIELD_CONFIDENCE` falls back to `medium` so an env typo cannot silently disable the filter. Both covered by tests.
- Tests now 22; verified the generated Gemini JSON schema includes the nested confidence object with a constrained `['high','medium','low']` enum, and that `to_public()` still emits exactly 10 keys.
- Caveat recorded for the interview: LLM self-reported confidence is not calibrated. This shifts the failure mode toward blank fields; it does not eliminate hallucination.
- Confidence approach REVERTED after user testing: the model rated reconstructed values as confidently legible, exactly the calibration failure flagged above. Removed `FieldConfidence`/`FieldConfidences`, `MIN_FIELD_CONFIDENCE`, and the prompt's rating rules; kept the transcribe-don't-reconstruct wording, which is independently useful.
- Replaced with self-consistency voting (previously identified as option 3): new pure module `app/consensus.py` runs a field-by-field majority vote across N independent extractions. Genuine readings repeat between runs, fabricated ones do not, so disagreement is treated as unreadable.
- `EXTRACTION_SAMPLES` (default 3, clamped 1-5) samples issued concurrently via ThreadPoolExecutor, so latency stays near a single call. Failed samples are dropped and consensus taken over the survivors; 502 only if every sample fails.
- Comparison is normalized (case, whitespace, trailing punctuation) so cosmetic variance is not counted as disagreement, but the most common ORIGINAL spelling is what gets returned.
- Temperature deliberately left at the model default — at temperature 0 all samples would be identical and the vote would detect nothing. Worth being able to explain this.
- `_resolve_extraction_outcome` now returns 422 whenever no field survives, not only when the model volunteered a readability complaint (per the requirement that fully inconsistent images are rejected as unclear).
- Two existing tests broke on that change: they set only an invalid field (`nidNumber="12345"`, `bloodGroup="XY"`), so after validation nulled it every field was empty and the new 422 fired. Added a valid `name` to each so they still test what they were written to test — that the validator nulls invalid values — rather than the new rule.
- Known interaction, left as-is: address mirroring runs after voting, so if one address survives the vote and the other does not, the survivor is mirrored into both. Correct for single-address cards (the majority), slightly generous for a two-address card with one illegible address.
- Tests now 27, including pure unit tests for the voting rules.

## 2026-07-23
- Added per-field consensus logging so the vote is visible in the terminal: one line per field with the verdict (KEPT / DROPPED / EMPTY), the agreement count against the sample total, plus an `is_nid` line and a `consensus result: N/10 field(s) kept` summary.
- Tension with the project's PII rule (metadata-only logs): showing "what the majority agreed on" means printing names, NID numbers and addresses. Resolved with two tiers — counts and verdicts always, values only under `CONSENSUS_LOG_VALUES` (default false, only an explicit truthy value enables it, so a typo fails closed). With it on, a disputed field also lists every competing reading and its count.
- Refactored the shared counting out of `vote()` into `_tally()`, which returns the leading value, its agreement count, and the values read. `vote()` is now a thin threshold check over it, so logging and the decision cannot drift apart by construction.
- Also logged the sample count up front and a warning when some samples failed and the vote ran over fewer than requested.
- Tests now 30: the default log contains the field names and counts but not the values, the flag adds them, and an invalid flag value stays off.
- Change request: prompt for a better photo whenever any field is unreadable, and refuse to return partial JSON at all once five or more fields are unreadable.
- Backend: `_resolve_extraction_outcome` now counts null fields and raises 422 above `config.MAX_UNREADABLE_FIELDS` (4), replacing the old "reject only if every field is null" rule. The count is taken after `to_public()`, so address mirroring has already run and a single-address card is not penalised. The count is also logged (a number, not the values).
- The model's own `readability_issue` is now a prefix to the standard message rather than a replacement for it: its complaints describe the problem but rarely tell the user what to do, and the requirement is that the user is asked to try again.
- Frontend: a warning banner inside the result panel lists the fields that could not be read and asks for a sharper, well-lit, glare-free photo. Shown for any missing field on a 200; the 422 case never reaches it and uses the existing error banner.
- Test fixtures had to change with the rule: seven tests built deliberately sparse extractions to isolate one field, which now trip the threshold. Added a `readable_extraction(**overrides)` helper that starts from the fully populated card and overrides only the field under test, so each test still exercises its own concern. Added boundary tests at four missing fields (200) and five (422).
- Tests now 32.
- The logs immediately surfaced two pre-existing behaviors worth knowing: address mirroring runs per sample (because `GeminiNIDExtraction` inherits the validator), so a lone address is already mirrored before the vote rather than after; and `_comparison_key` only strips edge punctuation, so "Md. Rahim Bhuiyan" and "Md Rahim Bhuiyan" count as disagreement. Left both unchanged — noted rather than silently retuned. Verified against the real observed failure: "Bhuiyan"/"Bhuiyan"/"Bhuiya" -> kept; three different readings -> null; sharp NID + smeared address -> NID kept, address dropped; nothing agreeing -> all null -> 422.
